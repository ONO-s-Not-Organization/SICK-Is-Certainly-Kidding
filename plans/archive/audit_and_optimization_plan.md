# 代码审计与逻辑优化方案

基于对当前项目的完整审计，我识别出了几个关键的逻辑缺陷和性能优化点。以下是详细的优化计划：

## 1. 核心逻辑优化

### 1.1 表格合并逻辑升级
- **现状**：`mergeRight` 和 `mergeDown` 仅能合并当前单元格与紧邻的一个单元格。
- **优化**：实现基于 `tableSelection` 范围的合并逻辑。
    - 将选定区域内的所有单元格内容合并到左上角单元格。
    - 计算选定区域的 `colSpan` 和 `rowSpan`。
    - 将选定区域内除左上角外的其他单元格标记为 `hidden`。

### 1.2 表格行列操作健壮性
- **现状**：插入或删除行列时直接进行数组切片，不考虑是否存在跨行或跨列的单元格，会导致表格结构错位或产生孤立的隐藏单元格。
- **优化**：在操作前检查受影响区域的 `colSpan` 和 `rowSpan`，动态调整跨度或禁止破坏性的删除。

### 1.3 类型安全强化
- **现状**：`Element.content` 使用了 `any` 类型。
- **优化**：定义具体的联合类型：
  ```typescript
  type ElementContent = string | ImageContent | TableData;
  ```
- 增强 `TableData` 的类型定义，确保嵌套结构的推断准确。

## 2. 性能优化

### 2.1 拖拽吸附逻辑重构
- **现状**：`DraggableElement.tsx` 在每次 `onDrag` 触发时都通过 `document.querySelector` 查询 DOM 元素尺寸。
- **优化**：
    - 在拖拽开始 (`onDragStart`) 时一次性获取并缓存所有参与对齐的元素边界信息。
    - 使用内存中的坐标计算代替 DOM 查询。

### 2.2 状态更新频率
- **现状**：部分表格调整会频繁触发全局 Store 更新。
- **优化**：评估对某些高频更新使用局部状态或节流处理。

## 3. 代码架构与 UX

### 3.1 工具函数提取
- 提取 `getColumnLabel` 等重复逻辑到 `src/utils/tableUtils.ts`。

### 3.2 配置验证
- 在 `importConfig` 中引入简单的模式验证，确保导入的 JSON 符合 `EditorState` 结构。

### 3.3 图层管理
- 在 `PropertyPanel` 中增加 `zIndex` 调整功能，替代目前固定的基于类型的排序。
